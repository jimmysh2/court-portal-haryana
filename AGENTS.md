## Identity of This Project

This is the **Naib Court Portal** — a judicial data entry and reporting system for Haryana courts.
It handles sensitive government data. Code quality, structural integrity, and localization accuracy are critical.

The project owner is **jimmysh2**. You work under their oversight. All code changes must be submitted via Pull Request, not pushed directly to `master`.

---

## Rule 1: Branch Before Any Code Change

**Before making any file edits**, you MUST create a feature branch.

```bash
git checkout master
git pull origin master
git checkout -b feature/your-task-description
```

If a feature branch already exists for the current task, check it out first:
```bash
git checkout feature/existing-branch-name
```

**Never commit directly to `master`.** Your push to `master` will be rejected by GitHub.

---

## Rule 2: Table & Column Changes

When asked to add, rename, or delete a table or column:
- You MAY manually edit `prisma/table-definitions.js` as it is the Single Source of Truth for table structures.
- Do NOT manually edit `prisma/seed-production.js` to change schema structures — it dynamically reads from `table-definitions.js`.
- Alternatively, you can use the Developer Dashboard UI, which triggers `scripts/auto-sync.js` to automatically re-generate `prisma/table-definitions.js` based on the live database.

If you edit `table-definitions.js` manually, you can apply your changes to a fresh database using `npm run seed:prod` or similar reset commands.

If a structural change is needed and the UI is not accessible, you must inform the user and ask them to make the change via the Developer Dashboard.

---

## Rule 3: Localization — No Hardcoded Text, Ever

Every user-facing string must be:
1. Added to `client/src/locales/translations.js` under **both** `en` and `hi` keys
2. Referenced in components using the `t()` hook from `useLanguage()`

```jsx
// ✅ CORRECT
const { t } = useLanguage();
return <button>{t('saveEntry')}</button>;

// ❌ WRONG — will be rejected in PR review
return <button>Save Entry</button>;
```

When adding a new translation key, always add both languages simultaneously:
```js
// In translations.js
en: { yourNewKey: "English Text" },
hi: { yourNewKey: "हिंदी पाठ" },
```

## Rule 4: Police Stations & Districts

Do NOT edit `Disrtrict_PS.csv` directly. This file is auto-generated when districts or police stations are modified via the System Management UI. If a police station change is requested, instruct the user to make the change through the System Management page.

## Rule 5: Push Workflow — Always via Pull Request

After completing a task:

```bash
# 1. Check what changed (auto-sync files will appear here)
git status

# 2. Stage everything
git add .

# 3. Commit with a clear, prefixed message
git commit -m "feat: Description of what was done"

# 4. Push the FEATURE BRANCH (not master)
git push origin feature/your-branch-name
```

Then open a Pull Request on GitHub for the project owner to review.

**Commit message prefixes:**
- `feat:` — new feature
- `fix:` — bug fix
- `ui:` — visual/styling change
- `lang:` — translation update
- `refactor:` — code restructuring
- `docs:` — documentation only
